54 /*!
55  * \defgroup ControlSystemGroup Control System
56  * \brief Contains control system elements
57  *
58  * The control system manages the time-dependent mapping between frames, such as
59  * the fixed computational frame (grid frame) and the inertial frame. The
60  * time-dependent parameters of the mapping are adjusted by a feedback control
61  * system in order to follow the dynamical evolution of objects such as horizons
62  * of black holes or surfaces of neutron stars. For example, in binary black
63  * hole simulations the map is typically a composition of maps that include
64  * translation, rotation, scaling, shape, etc.
65  * Each map under the governance of the control system has an associated
66  * time-dependent map parameter \f$\lambda(t)\f$ that is a piecewise Nth order
67  * polynomial. At discrete times (called reset times), the control system resets
68  * the Nth time derivative of \f$\lambda(t)\f$ to a new constant value, in order
69  * to minimize an error function \f$Q(t)\f$ that is specific to each map. At
70  * each reset time, the Nth derivative of \f$\lambda(t)\f$ is set to a function
71  * \f$U(t)\f$, called the control signal, that is determined by \f$Q(t)\f$ and
72  * its time derivatives and time integral. Note that \f$\lambda(t)\f$,
73  * \f$U(t)\f$, and \f$Q(t)\f$ can be vectors.
74  *
75  * The key components of the control system are:
76  * - FunctionsOfTime: each map has an associated FunctionOfTime that represents
77  * the map parameter \f$\lambda(t)\f$ and relevant time derivatives.
78  * - ControlError: each map has an associated ControlError that computes
79  * the error, \f$Q(t)\f$. Note that for each map, \f$Q(t)\f$ is defined to
80  * follow the convention that \f$dQ = -d \lambda\f$ as \f$Q \rightarrow 0\f$.
81  * - Averager: an averager can be used to average out the noise in the 'raw'
82  * \f$Q(t)\f$ returned by the ControlError.
83  * - Controller: the map controller computes the control signal \f$U(t)\f$ from
84  * \f$Q(t)\f$ and its time integral and time derivatives.
85  * The control is accomplished by setting the Nth derivative of
86  * \f$\lambda(t)\f$ to \f$U(t)\f$. Two common controllers are PID
87  * (proportional/integral/derivative)
88  * \f[U(t) = a_{0}\int_{t_{0}}^{t} Q(t') dt'+a_{1}Q(t)+a_{2}\frac{dQ}{dt}\f]
89  * or
90  * PND (proportional/N derivatives)
91  * \f[ U(t) = \sum_{k=0}^{N} a_{k} \frac{d^kQ}{dt^k} \f]
92  * The coefficients \f$ a_{k} \f$ in the computation of \f$U(t)\f$ are chosen
93  * at each time such that the error \f$Q(t)\f$ will be critically damped
94  * on a timescale of \f$\tau\f$ (the damping time),
95  * i.e. \f$Q(t) \propto e^{-t/\tau}\f$.
96  * - TimescaleTuner: each map has a TimescaleTuner that dynamically adjusts
97  * the damping timescale \f$\tau\f$ appropriately to keep the error \f$Q(t)\f$
98  * within some specified error bounds. Note that the reset time interval,
99  * \f$\Delta t\f$, is a constant fraction of this damping timescale,
100  * i.e. \f$\Delta t = \alpha \tau\f$ (empirically, we have found
101  * \f$\alpha=0.3\f$ to be a good choice).
102  *
103  *
104  * For additional details describing our control system approach, see Hemberger
105  * et al. 2012 (https://arxiv.org/abs/1211.6079).
106  */

126 /*!
127  * \defgroup DataBoxGroup DataBox
128  * \brief Documentation, functions, metafunctions, and classes necessary for
129  * using DataBox
130  *
131  * DataBox is a heterogeneous compile-time associative container with lazy
132  * evaluation of functions. DataBox can not only store data, but can also store
133  * functions that depend on other data inside the DataBox. The functions will be
134  * evaluated when the data they return is requested. The result is cached, and
135  * if a dependency of the function is modified the cache is invalidated.
136  *
137  * #### Simple and Compute Tags and Their Items
138  *
139  * The compile-time keys are `struct`s called tags, while the values are called
140  * items. Tags are quite minimal, containing only the information necessary to
141  * store the data and evaluate functions. There are two different types of tags
142  * that a DataBox can hold: simple tags and compute tags. Simple tags are for
143  * data that is inserted into the DataBox at the time of creation, while compute
144  * tags are for data that will be computed from a function when the compute item
145  * is retrieved. If a compute item is never retrieved from the DataBox then it
146  * is never evaluated.
147  *
148  * Simple tags must have a member type alias `type` that is the type of the data
149  * to be stored and a `static std::string name()` method that returns the name
150  * of the tag. Simple tags must inherit from `db::SimpleTag`.
151  *
152  * Compute tags must also have a `static std::string name()` method that returns
153  * the name of the tag, but they cannot have a `type` type alias. Instead,
154  * compute tags must have a static member function or static member function
155  * pointer named `function`. `function` can be a function template if necessary.
156  * The `function` must take all its arguments by `const` reference. The
157  * arguments to the function are retrieved using tags from the DataBox that the
158  * compute tag is in. The tags for the arguments are set in the member type
159  * alias `argument_tags`, which must be a `tmpl::list` of the tags corresponding
160  * to each argument. Note that the order of the tags in the `argument_list` is
161  * the order that they will be passed to the function. Compute tags must inherit
162  * from `db::ComputeTag`.
163  *
164  * Here is an example of a simple tag:
165  *
166  * \snippet Test_DataBox.cpp databox_tag_example
167  *
168  * and an example of a compute tag with a function pointer:
169  *
170  * \snippet Test_DataBox.cpp databox_compute_item_tag_example
171  *
172  * If the compute item's tag is inline then the compute item is of the form:
173  *
174  * \snippet Test_DataBox.cpp compute_item_tag_function
175  *
176  * Compute tags can also have their functions be overloaded on the type of its
177  * arguments:
178  *
179  * \snippet Test_DataBox.cpp overload_compute_tag_type
180  *
181  * or be overloaded on the number of arguments:
182  *
183  * \snippet Test_DataBox.cpp overload_compute_tag_number_of_args
184  *
185  * Compute tag function templates are implemented as follows:
186  *
187  * \snippet Test_DataBox.cpp overload_compute_tag_template
188  *
189  * Finally, overloading, function templates, and variadic functions can be
190  * combined to produce extremely generic compute tags. The below compute tag
191  * takes as template parameters a parameter pack of integers, which is used to
192  * specify several of the arguments. The function is overloaded for the single
193  * argument case, and a variadic function template is provided for the multiple
194  * arguments case. Note that in practice few compute tags will be this complex.
195  *
196  * \snippet Test_BaseTags.cpp compute_template_base_tags
197  *
198  * #### Subitems and Prefix Tags
199  *
200  * A simple or compute tag might also hold a collection of data, such as a
201  * container of `Tensor`s. In many cases you will want to be able to retrieve
202  * individual elements of the collection from the DataBox without having to
203  * first retrieve the collection. The infrastructure that allows for this is
204  * called *Subitems*. The subitems of the parent tag must refer to a subset of
205  * the data inside the parent tag, e.g. one `Tensor` in the collection. If the
206  * parent tag is `Parent` and the subitems tags are `Sub<0>, Sub<1>`, then when
207  * `Parent` is added to the DataBox, so are `Sub<0>` and `Sub<1>`. This means
208  * the retrieval mechanisms described below will work on `Parent`, `Sub<0>`, and
209  * `Sub<1>`.
210  *
211  * Subitems specify requirements on the tags they act on. For example, there
212  * could be a requirement that all tags with a certain type are to be treated as
213  * a Subitms. Let's say that the `Parent` tag holds a `Variables`, and
214  * `Variables` can be used with the Subitems infrastructure to add the nested
215  * `Tensor`s. Then all tags that hold a `Variables` will have their subitems
216  * added into the DataBox. To add a new type as a subitem the `db::Subitems`
217  * struct must be specialized. See the documentation of `db::Subitems` for more
218  * details.
219  *
220  * The DataBox also supports *prefix tags*, which are commonly used for items
221  * that are related to a different item by some operation. Specifically, say
222  * you have a tag `MyTensor` and you want to also have the time derivative of
223  * `MyTensor`, then you can use the prefix tag `dt` to get `dt<MyTensor>`. The
224  * benefit of a prefix tag over, say, a separate tag `dtMyTensor` is that prefix
225  * tags can be added and removed by the compute tags acting on the original tag.
226  * Prefix tags can also be composed, so a second time derivative would be
227  * `dt<dt<MyTensor>>`. The net result of the prefix tags infrastructure is that
228  * the compute tag that returns `dt<MyTensor>` only needs to know its input
229  * tags, it knows how to name its output based off that. In addition to the
230  * normal things a simple or a compute tag must hold, prefix tags must have a
231  * nested type alias `tag`, which is the tag being prefixed. Prefix tags must
232  * also inherit from `db::PrefixTag` in addition to inheriting from
233  * `db::SimpleTag` or `db::ComputeTag`.
234  *
235  * #### Creating a DataBox
236  *
237  * You should never call the constructor of a DataBox directly. DataBox
238  * construction is quite complicated and the helper functions `db::create` and
239  * `db::create_from` should be used instead. `db::create` is used to construct a
240  * new DataBox. It takes two typelists as explicit template parameters, the
241  * first being a list of the simple tags to add and the second being a list of
242  * compute tags to add. If no compute tags are being added then only the simple
243  * tags list must be specified. The tags lists should be passed as
244  * `db::create<db::AddSimpleTags<simple_tags...>,
245  * db::AddComputeTags<compute_tags...>>`. The arguments to `db::create` are the
246  * initial values of the simple tags and must be passed in the same order as the
247  * tags in the `db::AddSimpleTags` list. If the type of an argument passed to
248  * `db::create` does not match the type of the corresponding simple tag a static
249  * assertion will trigger. Here is an example of how to use `db::create`:
250  *
251  * \snippet Test_DataBox.cpp create_databox
252  *
253  * To create a new DataBox from an existing one use the `db::create_from`
254  * function. The only time a new DataBox needs to be created is when tags need
255  * to be removed or added. Like `db::create`, `db::create_from` also takes
256  * typelists as explicit template parameter. The first template parameter is the
257  * list of tags to be removed, which is passed using `db::RemoveTags`, second is
258  * the list of simple tags to add, and the third is the list of compute tags to
259  * add. If tags are only removed then only the first template parameter needs to
260  * be specified. If tags are being removed and only simple tags are being added
261  * then only the first two template parameters need to be specified. Here is an
262  * example of removing a tag or compute tag:
263  *
264  * \snippet Test_DataBox.cpp create_from_remove
265  *
266  * Adding a simple tag is done using:
267  *
268  * \snippet Test_DataBox.cpp create_from_add_item
269  *
270  * Adding a compute tag is done using:
271  *
272  * \snippet Test_DataBox.cpp create_from_add_compute_item
273  *
274  * #### Accessing and Mutating Items
275  *
276  * To retrieve an item from a DataBox use the `db::get` function. `db::get`
277  * will always return a `const` reference to the object stored in the DataBox
278  * and will also have full type information available. This means you are able
279  * to use `const auto&` when retrieving tags from the DataBox. For example,
280  * \snippet Test_DataBox.cpp using_db_get
281  *
282  * If you want to mutate the value of a simple item in the DataBox use
283  * `db::mutate`. Any compute item that depends on the mutated item will have its
284  * cached value invalidated and be recomputed the next time it is retrieved from
285  * the DataBox. `db::mutate` takes a parameter pack of tags to mutate as
286  * explicit template parameters, a `gsl::not_null` of the DataBox whose items
287  * will be mutated, an invokable, and extra arguments to forward to the
288  * invokable. The invokable takes the arguments passed from the DataBox by
289  * `const gsl::not_null` while the extra arguments are forwarded to the
290  * invokable. The invokable is not allowed to retrieve anything from the
291  * DataBox, so any items must be passed as extra arguments using `db::get` to
292  * retrieve them. For example,
293  *
294  * \snippet Test_DataBox.cpp databox_mutate_example
295  *
296  * In addition to retrieving items using `db::get` and mutating them using
297  * `db::mutate`, there is a facility to invoke an invokable with tags from the
298  * DataBox. `db::apply` takes a `tmpl::list` of tags as an explicit template
299  * parameter, will retrieve all the tags from the DataBox passed in and then
300  * invoke the invokable with the items in the tag list. Similarly,
301  * `db::mutate_apply` invokes the invokable but allows for mutating some of
302  * the tags. See the documentation of `db::apply` and `db::mutate_apply` for
303  * examples of how to use them.
304  *
305  * #### The Base Tags Mechanism
306  *
307  * Retrieving items by tags should not require knowing whether the item being
308  * retrieved was computed using a compute tag or simply added using a simple
309  * tag. The framework that handles this falls under the umbrella term
310  * *base tags*. The reason is that a compute tag can inherit from a simple tag
311  * with the same item type, and then calls to `db::get` with the simple tag can
312  * be used to retrieve the compute item as well. That is, say you have a compute
313  * tag `ArrayCompute` that derives off of the simple tag `Array`, then you can
314  * retrieve the compute tag `ArrayCompute` and `Array` by calling
315  * `db::get<Array>(box)`. The base tags mechanism requires that only one `Array`
316  * tag be present in the DataBox, otherwise a static assertion is triggered.
317  *
318  * The inheritance idea can be generalized further with what are called base
319  * tags. A base tag is an empty `struct` that inherits from `db::BaseTag`. Any
320  * simple or compute item that derives off of the base tag can be retrieved
321  * using `db::get`. Consider the following `VectorBase` and `Vector` tag:
322  *
323  * \snippet Test_BaseTags.cpp vector_base_definitions
324  *
325  * It is possible to retrieve `Vector<1>` from the DataBox using
326  * `VectorBase<1>`. Most importantly, base tags can also be used in compute tag
327  * arguments, as follows:
328  *
329  * \snippet Test_BaseTags.cpp compute_template_base_tags
330  *
331  * As shown in the code example, the base tag mechanism works with function
332  * template compute tags, enabling generic programming to be combined with the
333  * lazy evaluation and automatic dependency analysis offered by the DataBox. To
334  * really demonstrate the power of base tags, let's also have `ArrayComputeBase`
335  * inherit from a simple tag `Array`, which inherits from a base tag `ArrayBase`
336  * as follows:
337  *
338  * \snippet Test_BaseTags.cpp array_base_definitions
339  *
340  * To start, let's create a DataBox that holds a `Vector<0>` and an
341  * `ArrayComputeBase<0>` (the concrete tag must be used when creating the
342  * DataBox, not the base tags), retrieve the tags using the base tag mechanism,
343  * including mutating `Vector<0>`, and then verifying that the dependencies are
344  * handled correctly.
345  *
346  * \snippet Test_BaseTags.cpp base_simple_and_compute_mutate
347  *
348  * Notice that we are able to retrieve `ArrayComputeBase<0>` with `ArrayBase<0>`
349  * and `Array<0>`. We were also able to mutate `Vector<0>` using
350  * `VectorBase<0>`.
351  *
352  * We can even remove tags using their base tags with `db::create_from`:
353  *
354  * \snippet Test_BaseTags.cpp remove_using_base
355  *
356  * The base tags infrastructure even works with Subitems. Even if you mutate the
357  * subitem of a parent using a base tag, the appropriate compute item caches
358  * will be invalidated.
359  *
360  * \note All of the base tags infrastructure works for `db::get`, `db::mutate`,
361  * `db::apply` and `db::mutate_apply`.
362  */

464 /*!
465  * \defgroup ParallelGroup Parallelization
466  * \brief Functions, classes and documentation related to parallelization and
467  * Charm++
468 
469 SpECTRE builds a layer on top of Charm++ that performs various safety checks and
470 initialization for the user that can otherwise lead to difficult-to-debug
471 undefined behavior. The central concept is what is called a %Parallel
472 Component. A %Parallel Component is a struct with several type aliases that
473 is used by SpECTRE to set up the Charm++ chares and allowed communication
474 patterns. %Parallel Components are input arguments to the compiler, which then
475 writes the parallelization infrastructure that you requested for the executable.
476 There is no restriction on the number of %Parallel Components, though
477 practically it is best to have around 10 at most.
478 
479 Here is an overview of what is described in detail in the sections below:
480 
481 - Metavariables: Provides high-level configuration to the compiler, e.g. the
482  physical system to be simulated.
483 - Phase: Defines distinct simulation phases separated by a global
484  synchronization point, e.g. `Initialize`, `Evolve` and `Exit`.
485 - Algorithm: In each phase, iterates over a list of actions until the current
486  phase ends.
487 - %Parallel component: Maintains and executes its algorithm.
488 - Action: Performs a computational task, e.g. evaluating the right hand side of
489  the time evolution equations. May require data to be received from another
490  action potentially being executed on a different core or node.
491 
492 ### The Metavariables Class
493 
494 SpECTRE takes a different approach to input options passed to an executable than
495 is common. SpECTRE not only reads an input file at runtime but also has many
496 choices made at compile time. The compile time options are specified by what is
497 referred to as the metavariables. What exactly the metavariables struct
498 specifies depends somewhat on the executable, but all metavariables structs must
499 specify the following:
500 
501 - `help`: a `static constexpr OptionString` that will be printed as part of the
502  help message. It should describe the executable and basic usage of it, as well
503  as any non-standard options that must be specified in the metavariables and
504  their current values. An example of a help string for one of the testing
505  executables is:
506  \snippet Test_AlgorithmCore.cpp help_string_example
507 - `component_list`: a `tmpl::list` of the parallel components (described below)
508  that are to be created. Most evolution executables will have the
509  `DgElementArray` parallel component listed. An example of a `component_list`
510  for one of the test executables is:
511  \snippet Test_AlgorithmCore.cpp component_list_example
512 - `using const_global_cache_tag_list` is set to a (possibly empty) `tmpl::list`
513  of CacheTags that are needed by the metavariables.
514 - `Phase`: an `enum class` that must contain at least `Initialization` and
515  `Exit`. Phases are described in the next section.
516 - `determine_next_phase`: a static function with the signature
517  \code
518  static Phase determine_next_phase(
519  const Phase& current_phase,
520  const Parallel::CProxy_ConstGlobalCache<EvolutionMetavars>& cache_proxy)
521  noexcept;
522  \endcode
523  What this function does is described below in the discussion of phases.
524 
525 There are also several optional members:
526 
527 - `input_file`: a `static constexpr OptionString` that is the default name of
528  the input file that is to be read. This can be overridden at runtime by
529  passing the `--input-file` argument to the executable.
530 - `ignore_unrecognized_command_line_options`: a `static constexpr bool` that
531  defaults to `false`. If set to `true` then unrecognized command line options
532  are ignored. Ignoring unrecognized options is generally only necessary for
533  tests where arguments for the testing framework, Catch, are passed to the
534  executable.
535 
536 ### Phases of an Execution
537 
538 Global synchronization points, where all cores wait for each other, are
539 undesirable for scalability reasons. However, they are sometimes inevitable for
540 algorithmic reasons. That is, in order to actually get a correct solution you
541 need to have a global synchronization. SpECTRE executables can have multiple
542 phases, where after each phase a global synchronization occurs. By global
543 synchronization we mean that no parallel components are executing or have more
544 tasks to execute: everything is waiting on a task to perform.
545 
546 Every executable must have at least two phases, `Initialization` and `Exit`. The
547 next phase is decided by the static member function `determine_next_phase` in
548 the metavariables. Currently this function has access to the phase that is
549 ending, and also the global cache. In the future we will add support for
550 receiving data from various components to allow for more complex decision
551 making. Here is an example of a `determine_next_phase` function and the `Phase`
552 enum class:
553 \snippet Test_AlgorithmCore.cpp determine_next_phase_example
554 
555 In contrast, an evolution executable might have phases `Initialization`,
556 `SetInitialData`, `Evolve`, and `Exit`, but have a similar `switch` or `if-else`
557 logic in the `determine_next_phase` function. The first phase that is entered is
558 always `Initialization`. During the `Initialization` phase the `initialize`
559 function is called on all parallel components. Once all parallel components'
560 `initialize` function is complete, the next phase is determined and the
561 `execute_next_phase` function is called after on all the parallel components.
562 
563 At the end of an execution the `Exit` phase has the executable wait to make sure
564 no parallel components are performing or need to perform any more tasks, and
565 then exits. An example where this approach is important is if we are done
566 evolving a system but still need to write data to disk. We do not want to exit
567 the simulation until all data has been written to disk, even though we've
568 reached the final time of the evolution.
569 
570 ### The Algorithm
571 
572 Since most numerical algorithms repeat steps until some criterion such as the
573 final time or convergence is met, SpECTRE's parallel components are designed to
574 do such iterations for the user. An Algorithm executes an ordered list of
575 actions until one of the actions cannot be evaluated, typically because it is
576 waiting on data from elsewhere. When an algorithm can no longer evaluate actions
577 it passively waits by handing control back to Charm++. Once an algorithm
578 receives data, typically done by having another parallel component call the
579 `receive_data` function, the algorithm will try again to execute the next
580 action. If the algorithm is still waiting on more data then the algorithm will
581 again return control to Charm++ and passively wait for more data. This is
582 repeated until all required data is available. The actions that are iterated
583 over by the algorithm are called iterable actions and are described below.
584 
585 \note
586 Currently all Algorithms must execute the same actions (described below) in all
587 phases. This restriction is also planned on being relaxed if the need arises.
588 
589 ### %Parallel Components
590 
591 Each %Parallel Component struct must have the following type aliases:
592 1. `using chare_type` is set to one of:
593  1. `Parallel::Algorithms::Singleton`s have one object in the entire execution
594  of the program.
595  2. `Parallel::Algorithms::Array`s hold zero or more elements, each of which
596  is an object distributed to some core. An array can grow and shrink in
597  size dynamically if need be and can also be bound to another array. A
598  bound array has the same number of elements as the array it is bound to,
599  and elements with the same ID are on the same core. See Charm++'s chare
600  arrays for details.
601  3. `Parallel::Algorithms::Group`s are arrays with
602  one element per core which are not able to be moved around between
603  cores. These are typically useful for gathering data from array elements
604  on their core, and then processing or reducing the data further. See
605  [Charm++'s](http://charm.cs.illinois.edu/help) group chares for details.
606  4. `Parallel::Algorithms::Nodegroup`s are similar to
607  groups except that there is one element per node. For Charm++ SMP (shared
608  memory parallelism) builds, a node corresponds to the usual definition of
609  a node on a supercomputer. However, for non-SMP builds nodes and cores are
610  equivalent. We ensure that all entry method calls done through the
611  Algorithm's `simple_action` and `receive_data` functions are
612  threadsafe. User controlled threading is possible by calling the non-entry
613  method member function `threaded_action`.
614 2. `using metavariables` is set to the Metavariables struct that stores the
615  global metavariables. It is often easiest to have the %Parallel
616  Component struct have a template parameter `Metavariables` that is the
617  global metavariables struct. Examples of this technique are given below.
618 3. `using action_list` is set to a `tmpl::list` of the %Actions (described
619  below) that the Algorithm running on the %Parallel Component executes. The
620  %Actions are executed in the order that they are given in the `tmpl::list`.
621 4. `using initial_databox` is set to the type of the DataBox that will be passed
622  to the first Action of the `action_list`. Typically it is the output of some
623  simple action called during the `Initialization` Phase.
624 5. `using options` is set to a (possibly empty) `tmpl::list` of the option
625  structs. The options are read in from the input file specified in the main
626  `Metavariables` struct. After being read in they are passed to the
627  `initialize` function of the parallel component, which is described below.
628 6. `using const_global_cache_tag_list` is set to a `tmpl::list` of CacheTags
629  that are required by the parallel component. This is usually obtained from
630  the `action_list` using the `Parallel::get_const_global_cache_tags`
631  metafunction.
632 
633 \note Array parallel components must also specify the type alias `using
634 array_index`, which is set to the type that indexes the %Parallel Component
635 Array. Charm++ allows arrays to be 1 through 6 dimensional or be indexed by a
636 custom type. The Charm++ provided indexes are wrapped as
637 `Parallel::ArrayIndex1D` through `Parallel::ArrayIndex6D`. When writing custom
638 array indices, the [Charm++ manual](http://charm.cs.illinois.edu/help) tells you
639 to write your own `CkArrayIndex`, but we have written a general implementation
640 that provides this functionality; all that you need to provide is a
641 plain-old-data
642 ([POD](http://en.cppreference.com/w/cpp/concept/PODType)) struct of the size of
643 at most 3 integers.
644 
645 %Parallel Components have a static `initialize` function that is used
646 effectively as the constructor of the components. The signature of the
647 initialize functions must be:
648 \code
649 static void initialize(
650  Parallel::CProxy_ConstGlobalCache<metavariables>& global_cache, opts...);
651 \endcode
652 The `initialize` function is called by the Main %Parallel Component when
653 the execution starts and will typically call a simple %Action
654 to set up the initial state of the Algorithm, similar to what a constructor
655 does for classes. The `initialize` function also receives arguments that
656 are read from the input file which were specified in the `options` typelist
657 described above. The options are usually used to initialize the %Parallel
658 Component's DataBox, or even the component itself. An example of initializing
659 the component itself would be using the value of an option to control the size
660 of the %Parallel Component Array. The `initialize` functions of different
661 %Parallel Components are called in random order and so it is not safe to have
662 them depend on each other.
663 
664 Each parallel component must also decide what to do in the different phases of
665 the execution. This is controlled by an `execute_next_phase` function with
666 signature:
667 \code
668 static void execute_next_phase(
669  const typename metavariables::Phase next_phase,
670  const Parallel::CProxy_ConstGlobalCache<metavariables>& global_cache);
671 \endcode
672 The `determine_next_phase` function in the Metavariables determines the next
673 phase, after which the `execute_next_phase` function gets called. The
674 `execute_next_phase` function determines what the %Parallel Component should do
675 during the next phase. For example, it may simply call `perform_algorithm`, call
676 a series of simple actions, perform a reduction over an Array, or not do
677 anything at all. Note that `perform_algorithm` performs the same actions (the
678 ones in `action_list`) no matter what Phase it is called in.
679 
680 An example of a singleton %Parallel Component is:
681 \snippet Test_AlgorithmParallel.cpp singleton_parallel_component
682 
683 An example of an array %Parallel Component is:
684 \snippet Test_AlgorithmParallel.cpp array_parallel_component
685 Elements are inserted into the Array by using the Charm++ `insert` member
686 function of the CProxy for the array. The `insert` function is documented in
687 the Charm++ manual. In the above Array example `array_proxy` is a `CProxy` and
688 so all the documentation for Charm++ array proxies applies. SpECTRE always
689 creates empty Arrays with the constructor and requires users to insert however
690 many elements they want and on which cores they want them to be placed. Note
691 that load balancing calls may result in Array elements being moved.

693 ### %Actions
694 
695 For those familiar with Charm++, actions should be thought of as effectively
696 being entry methods. They are functions that can be invoked on a remote object
697 (chare/parallel component) using a `CProxy` (see the [Charm++
698 manual](http://charm.cs.illinois.edu/help)), which is retrieved from the
699 ConstGlobalCache using the parallel component struct and the
700 `Parallel::get_parallel_component()` function. %Actions are structs with a
701 static `apply` method and come in three variants: simple actions, iterable
702 actions, and reduction actions. One important thing to note
703 is that actions cannot return any data to the caller of the remote method.
704 Instead, "returning" data must be done via callbacks or a callback-like
705 mechanism.
706 
707 The simplest signature of an `apply` method is for iterable actions:
708 \snippet Test_AlgorithmCore.cpp apply_iterative
709 The return type is discussed at the end of each section describing a particular
710 type of action. Simple actions can have additional arguments but must have at
711 least the arguments shown above. Reduction actions must have the above arguments
712 and an argument taken by value that is of the type the reduction was made over.
713 The `db::DataBox` should be thought of as the member data of the parallel
714 component while the actions are the member functions. The combination of a
715 `db::DataBox` and actions allows building up classes with arbitrary member data
716 and methods using template parameters and invocation of actions. This approach
717 allows us to eliminate the need for users to work with Charm++'s interface
718 files, which can be error prone and difficult to use.
719 
720 The ConstGlobalCache is passed to each action so that the action has access
721 to global data and is able to invoke actions on other parallel components. The
722 `ParallelComponent` template parameter is the tag of the parallel component that
723 invoked the action. A proxy to the calling parallel component can then be
724 retrieved from the ConstGlobalCache. The remote entry method invocations are
725 slightly different for different types of actions, so they will be discussed
726 below. However, one thing that is disallowed for all actions is calling an
727 action locally from within an action on the same parallel component.
728 Specifically,

733 local (currently executing) parallel component. See the [Charm++
734 manual](http://charm.cs.illinois.edu/help) for more information.

736 parallel component by getting your own parallel component from the
737 ConstGlobalCache (`Parallel::get_parallel_component<ParallelComponent>(cache)`).

764 #### 1. Simple %Actions
765 
766 Simple actions are designed to be called in a similar fashion to member
767 functions of classes. They are the direct analog of entry methods in Charm++
768 except that the member data is stored in the `db::DataBox` that is passed in as
769 the first argument. There are a couple of important things to note with simple
770 actions:
771 
772 1. A simple action must return void but can use `db::mutate` to change values
773  of items in the DataBox if the DataBox is taken as a non-const reference.
774  There is one exception: if the input DataBox is empty, then the
775  simple action can return a DataBox of type `initial_databox`. That is, an
776  action taking an empty DataBox and returning the `initial_databox` is
777  effectively constructing the DataBox in its initial state.
778 2. A simple action is instantiated once for an empty
779  `db::DataBox<tmpl::list<>>`, once for a DataBox of type
780  `initial_databox` (listed in the parallel component), and once for each
781  returned DataBox from the iterable actions in the `action_list` in the
782  parallel component. In some cases you will need specific items to be in the
783  DataBox otherwise the action won't compile. To restrict which DataBoxes can
784  be passed you should use `Requires` in the action's `apply` function
785  template parameter list. For example,
786  \snippet Test_AlgorithmCore.cpp requires_action
787  where the conditional checks if any element in the parameter pack `DbTags` is
788  `CountActionsCalled`.

820 components has been received. For example, it may check that all data from

1061  * providing a tighter tolerance with `approx`, we avoid having to redefine the
1062  * tolerance in every test.
1063  *
1064  * ### Attributes
1065  *
1066  * Attributes allow you to modify properties of the test. Attributes are
1067  * specified as follows:
1068  * \code
1069  * // [[TimeOut, 10]]
1070  * // [[OutputRegex, The error message expected from the test]]
1071  * SPECTRE_TEST_CASE("Unit.Blah", "[Unit]") {
1072  * \endcode
1073  *
1074  * Available attributes are:
1075  *
1076  * <table class="doxtable">
1077  * <tr>
1078  * <th>Attribute </th><th>Description </th>
1079  * </tr>
1080  * <tr>
1081  * <td>TimeOut </td>
1082  * <td>override the default timeout and set the timeout to N seconds. This
1083  * should be set very sparingly since unit tests are designed to be
1084  * short. If your test is too long you should consider testing smaller
1085  * portions of the code if possible, or writing an integration test instead.
1086  * </td>
1087  * </tr>
1088  * <tr>
1089  * <td>OutputRegex </td>
1090  * <td>
1091  * When testing failure modes the exact error message must be tested, not
1092  * just that the test failed. Since the string passed is a regular
1093  * expression you must escape any regex tokens. For example, to match
1094  * `some (word) and` you must specify the string `some \(word\) and`.
1095  * If your error message contains a newline, you can match it using the
1096  * dot operator `.`, which matches any character.
1097  * </td>
1098  * </tr>
1099  * </table>
1100  *
1101  * \example
1102  * \snippet Test_H5.cpp willfail_example_for_dev_doc
1103  *
1104  * ### Testing static assert
1105  *
1106  * You are able to test that a `static_assert` is being triggered using
1107  * the compilation failure test framework. When creating a new `static_assert`
1108  * test you must be sure to not have it in the same file as the runtime tests
1109  * since the file will not compile. The new file, say
1110  * `Test_StaticAssertDataBox.cpp` must be added to the
1111  * `SPECTRE_COMPILATION_TESTS` CMake variable, not `SPECTRE_TESTS`. Here is
1112  * an example of how to write a compilation failure test:
1113  *
1114  * \snippet TestCompilationFramework.cpp compilation_test_example
1115  *
1116  * Each individual test must be inside an `#%ifdef COMPILATION_TEST_.*` block
1117  * and each compilation test `cpp` file must contain
1118  * `FILE_IS_COMPILATION_TEST` outside of any `#%ifdef`s and at the end of
1119  * the file.
1120  *
1121  * Specific compiler versions can be specified for which the regex changes.
1122  * That is, the compiler version specified and all versions newer than that
1123  * will use the regex, until a newer compiler version is specified. For
1124  * example, see the below code prints a different static_assert for pre-GCC 6
1125  * and GCC 6 and newer.
1126  *
1127  * \snippet TestCompilationFramework.cpp gnu_versions_example
1128  *
1129  * ### Debugging Tests in GDB or LLDB
1130  *
1131  * Several tests fail intentionally at the executable level to test error
1132  * handling like ASSERT statements in the code. CTest is aware of which
1133  * should fail and passes them. If you want to debug an individual test
1134  * in a debugger you need to run a single test
1135  * using the %RunTests executable (in dg-charm-build/bin/RunTests) you
1136  * must specify the name of the test as the first argument. For example, if you
1137  * want to run just the "Unit.Gradient" test you can run
1138  * `./bin/RunTests Unit.Gradient`. If you are using a debugger launch the
1139  * debugger, for example if you're using LLDB then run `lldb ./bin/RunTests`
1140  * and then to run the executable inside the debugger use `run Unit.Gradient`
1141  * inside the debugger.
1142  */

1144 /*!
1145  * \defgroup TimeGroup Time
1146  * \brief Code related to the representation of time during simulations.
1147  *
1148  * The time covered by a simulation is divided up into a sequence of
1149  * adjacent, non-overlapping (except at endpoints) intervals referred
1150  * to as "slabs". The boundaries between slabs can be placed at
1151  * arbitrary times. Slabs, as represented in the code as the Slab
1152  * class, provide comparison operators comparing slabs agreeing with
1153  * the definition as a sequence of intervals. Slabs that do not
1154  * jointly belong to any such sequence should not be compared.
1155  *
1156  * The specific time is represented by the Time class, which encodes
1157  * the slab containing the time and the fraction of the slab that has
1158  * elapsed as an exact rational. Times are comparable according to
1159  * their natural time ordering, except for times belonging to
1160  * incomparable slabs.
1161  *
1162  * Differences in time within a slab are represented as exact
1163  * fractions of that slab by the TimeDelta class. TimeDeltas are only
1164  * meaningful within a single slab, with the exception that the ratio
1165  * of objects with different slabs may be taken, resulting in an
1166  * inexact floating-point result. Longer intervals of time are
1167  * represented using floating-point values.
1168  */
